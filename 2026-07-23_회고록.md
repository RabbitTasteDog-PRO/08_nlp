# 정제 및 정규화

> 2026년 7월 23일 회고

정제(cleansing)는 분석에 불필요하거나 오류가 있는 요소를 제거·마스킹하는 작업이다. 예를 들어 중복 공백, 너무 짧은 토큰, 불용어, 이메일·URL 같은 민감 정보를 처리한다. 반면 정규화(normalization)는 의미가 같은 서로 다른 표현을 하나의 기준형으로 통일하는 작업이다. 대소문자, 전각·반각 문자, 유니코드 표현, `United Kingdom`과 `UK`처럼 같은 대상을 가리키는 표현을 맞추는 것이 예시다.

두 과정의 목적은 모델이 같은 의미의 입력을 일관되게 처리하도록 만드는 것이다. 다만 감성 분석의 부정어처럼 중요한 정보까지 제거하지 않도록, 데이터와 분석 목적에 따라 규칙을 설계해야 한다.

오늘은 NLP 텍스트 전처리의 기본 과정을 직접 확인했다.

## 01. 자연어 처리 개요

자연어 처리는 사람이 사용하는 언어를 컴퓨터가 처리할 수 있는 형태로 바꾸고, 그 안의 의미와 구조를 분석하는 분야라는 점을 정리했다. 감성 분석, 번역, 검색, 챗봇, 문서 분류처럼 적용 분야는 다양하지만, 어떤 과업이든 신뢰할 수 있는 코퍼스와 목적에 맞는 데이터셋이 먼저 필요하다.

말뭉치, 문장, 단어, 토큰, 형태소, 불용어는 비슷해 보이지만 서로 다른 분석 단위다. 특히 한국어는 조사와 어미가 단어에 붙기 때문에 띄어쓰기 단위만으로 의미를 분석하기 어렵고, 형태소 단위 분석이 필요할 수 있다는 점을 이해했다. 또한 문장의 의미는 단어 자체뿐 아니라 문맥, 문장 구조, 담화 상황에 따라 달라진다. 같은 단어라도 문맥에 따라 품사가 달라지고, 문장 하나만으로는 화자의 의도나 지시 대상을 완전히 판단하기 어렵다.

모델은 텍스트 자체를 바로 계산하지 못하므로 토큰이나 문장 같은 표현 단위를 숫자 벡터로 바꿔야 한다. 전통적인 BoW·TF-IDF 방식과 Word2Vec 같은 임베딩, 문맥을 반영하는 Transformer 임베딩은 각각 표현하는 정보와 장단점이 다르다. 결국 데이터 특성, 과업 목표, 계산 비용을 고려해 표현 방법을 선택해야 한다.

## 02. 텍스트 전처리

### 토큰화와 문장 분리

토큰화에서는 같은 문장도 방식에 따라 결과가 크게 달라진다는 점이 인상적이었다. 정규표현식은 `Don't`, `well-being`처럼 원문 형태를 비교적 잘 보존했고, `WordPunctTokenizer`는 구두점을 기준으로 세밀하게 분리했다. `word_tokenize`는 영어 문법 관례에 따라 `Don't`를 `Do`, `n't`로 나누었다. BERT의 WordPiece 토크나이저는 `unhappiness`를 `un`, `##ha`, `##pp`, `##iness`처럼 서브워드 단위로 나누었다. 토큰화에는 하나의 정답이 있는 것이 아니라, 검색·문법 분석·모델 입력 등 목적에 맞는 기준이 필요하다는 것을 알게 됐다.

아래 코드는 정규표현식, `WordPunctTokenizer`, `word_tokenize`의 차이를 비교한 실습이다.

```python
import re
from nltk.tokenize import WordPunctTokenizer, word_tokenize

comparison_text = "Don't hesitate to use well-being practices for self-care."

regex_pattern = r"[A-Za-z]+(?:[-'][A-Za-z]+)*|[^\w\s]"
regex_tokens = re.findall(regex_pattern, comparison_text)
word_punct_tokens = WordPunctTokenizer().tokenize(comparison_text)
nltk_word_tokens = word_tokenize(comparison_text)

print("정규식:", regex_tokens)
print("WordPunctTokenizer:", word_punct_tokens)
print("word_tokenize:", nltk_word_tokens)
```

다음은 BERT가 문장을 서브워드로 나누고, 특수 토큰을 추가하는 과정을 확인한 코드다.

```python
from transformers import AutoTokenizer

tokenizer = AutoTokenizer.from_pretrained("bert-base-uncased")
subword_text = "unhappiness and reusable tokenizers"

subword_tokens = tokenizer.tokenize(subword_text)
subword_ids = tokenizer.convert_tokens_to_ids(subword_tokens)
special_tokens = tokenizer.convert_ids_to_tokens(
    tokenizer.encode(subword_text, add_special_tokens=True)
)

print("원본:", subword_text)
print("서브워드 토큰:", subword_tokens)
print("서브워드 ID:", subword_ids)
print("특수 토큰 포함:", special_tokens)
```

한국어 문장 분리에서는 NLTK보다 KSS가 더 자연스러운 결과를 냈다. 특히 인용문이 포함된 문장에서 NLTK는 문장을 잘못 나눴지만, KSS는 한국어 종결 표현과 인용 구조를 고려해 문장을 더 정확히 처리했다. 언어마다 띄어쓰기, 형태소, 문장부호 규칙이 다르므로 언어별 전처리 도구가 필요한 이유도 이해했다.

```python
from nltk.tokenize import sent_tokenize
import kss

korean_text = (
    '오늘은 비가 온다. 우산을 챙겼다. '
    '"내일도 비가 올까?"라고 친구가 물었다. 그래도 우리는 산책을 갔다.'
)

nltk_korean_sentences = sent_tokenize(korean_text)
kss_korean_sentences = kss.split_sentences(korean_text)

print("[NLTK]", nltk_korean_sentences)
print("[KSS]", kss_korean_sentences)
```

### 정제와 정규화

정규표현식은 전화번호·URL·특수문자·표현 통일처럼 규칙이 명확한 데이터를 처리할 때 유용하다. 사전학습 모델이 문맥 이해를 잘하더라도, 형식이 일정한 정보의 검증·마스킹·정규화까지 완벽하게 대신하지는 못한다. 실제 NLP는 정규표현식, 토크나이저, 언어별 분석기, 모델을 함께 쓰는 과정이라는 생각이 들었다.

오늘 가장 헷갈렸던 부분은 파이썬의 함수 전달과 콜백이었다. `key=get_term_length`는 함수를 실행하는 것이 아니라 `sorted()`에 넘기는 것이고, `sorted()`가 각 항목을 넣어 나중에 호출한다. 문법만 보면 인자가 어디서 들어오는지 보이지 않아 답답했지만, `함수이름`은 함수 전달, `함수이름(...)`은 함수 실행이라는 규칙을 정리했다. `lambda`도 특별한 마법이 아니라 이름 없는 함수를 한 줄로 만든 표현이라는 점을 이해했다.

용어 정규화 실습에서는 정규표현식과 콜백 함수를 함께 사용했다. 긴 표현을 먼저 치환하기 위해 `sorted()`에 `get_term_length`를 정렬 기준 함수로 전달했다.

```python
import re


def get_term_length(item):
    return len(item[0])


def normalize_terms(text, mapping):
    ordered_terms = sorted(
        mapping.items(),
        key=get_term_length,
        reverse=True,
    )

    normalized = text

    for source, target in ordered_terms:
        pattern = rf"\b{re.escape(source)}\b"
        normalized = re.sub(
            pattern,
            target,
            normalized,
            flags=re.IGNORECASE,
        )

    return normalized


raw_text = "The United Kingdom and UK have a long history together."
term_mapping = {"United Kingdom": "UK"}

normalized_text = normalize_terms(raw_text, term_mapping)

print("원문:", raw_text)
print("정규화:", normalized_text)
```

정제 단계에서는 빈도가 너무 낮거나 길이가 짧은 토큰, 불필요한 불용어를 목적에 따라 제외할 수 있다. 다만 감성 분석에서는 `not`, `never` 같은 부정어를 불용어로 제거하면 문장의 의미가 반대로 바뀔 수 있으므로 보호해야 한다. 이메일과 URL은 제거하기보다 `__EMAIL__`, `__URL__`처럼 마스킹하면 개인정보를 보호하면서 해당 정보가 등장했다는 특징은 남길 수 있다.

한국어 불용어 처리에서는 형태소 단위가 중요했다. `이를 통해`처럼 여러 토큰으로 나뉘는 표현은 문자열 하나와 비교하는 방식으로 제거할 수 없으므로, 연속된 토큰 구간이 구문 불용어와 일치하는지 확인해야 한다. 또한 Okt는 Java 기반 분석기를 사용하므로 실행 환경에서 JDK와 `JAVA_HOME` 설정이 필요하다는 점도 확인했다.

문자열 정규화도 확인했다. NFC는 분리된 한글 음절을 완성형으로 통일하고, NFKC는 `ＡＩ` 같은 전각 문자와 `①` 같은 호환 문자를 `AI`, `1`처럼 일반적인 형태로 바꾼다. 눈에는 비슷해 보여도 유니코드 값이 다르면 검색과 매칭에서 문제가 생길 수 있기 때문에, 텍스트 전처리 단계에서 문자 정규화가 필요하다는 점을 배웠다.

```python
import unicodedata

raw_text = "ＡＩ   모델\t가격은 ① 10,000원이다."

nfc_text = unicodedata.normalize("NFC", raw_text)
nfkc_text = unicodedata.normalize("NFKC", raw_text)

print("원문:", raw_text)
print("NFC:", nfc_text)
print("NFKC:", nfkc_text)

for char in raw_text:
    print(repr(char), f"U+{ord(char):04X}", unicodedata.name(char, "이름 없음"))
```
