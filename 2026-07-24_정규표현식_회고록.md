# 2026-07-24 NLP 학습 회고

오늘은 텍스트 전처리에서 단어를 어떻게 분석하고 정리하는지, 그리고 정규 표현식으로 원하는 문자열을 찾고 바꾸는 방법을 연결해서 학습했다.

게임을 만들 때는 주로 클라이언트 기능과 화면·상호작용을 구현했고, 백엔드까지 깊게 맡는 경우가 많지 않았다. 그래서 입력 문자열을 규칙으로 해석하거나, 데이터를 정제해서 다음 단계로 넘기는 작업을 직접 고민할 일이 적었다. 오늘 학습은 평소에는 그냥 텍스트로 보던 문장을 컴퓨터가 처리할 수 있는 단위와 규칙으로 바라보는 첫걸음이었다.

## 오늘 이해한 내용

### 1. 형태소 분석과 표제어 추출은 다르다

처음에는 형태소 분석이 `running -> run`처럼 단어를 원래 형태로 돌리는 작업이라고 생각했다. 하지만 형태소 분석은 문장을 의미를 가진 작은 단위로 나누고, 각 단위의 품사를 파악하는 작업이다.

예를 들어 `아버지가 방에 들어가신다`는 대략 `아버지/가`, `방/에`, `들어가/시/ㄴ다`로 나눌 수 있다. 이때 `들어가`는 어간이고, `시`는 높임을 나타내며, `ㄴ다`는 문장을 끝내는 어미이다.

반대로 표제어 추출(lemmatization)은 활용형을 사전의 기본형에 가깝게 바꾸는 작업이다. 영어에서는 `ran`, `running`을 `run`으로 바꾸고, 한국어에서는 `기다렸다`를 `기다리다`로 정규화하는 것이 예시다.

### 2. Okt의 `stem=True`

Okt는 한국어 문장을 형태소로 분석하는 도구이다. `morphs()`는 형태소를 나누고, `pos()`는 형태소와 품사를 함께 보여 주며, `nouns()`는 명사만 추출한다.

`stem=True`는 동사와 형용사의 활용형을 기본형에 가깝게 정규화하는 옵션이다. 따라서 `기다렸다`처럼 활용된 표현을 `기다리다`로 다룰 수 있다. 이는 영어의 표제어 추출과 비슷한 목적이지만, 한국어 형태소 분석 과정에서 용언을 정규화한다는 점을 기억해야 한다.

게임 클라이언트 개발에서는 이런 분석 결과를 직접 설계할 일이 많지 않아 `stem`이나 형태소라는 말이 처음에는 낯설었다. 하지만 채팅, 퀘스트 설명, 사용자 검색어처럼 게임 안에서도 텍스트가 쌓이는 기능을 생각해 보니, 텍스트를 정리하고 비교하기 전에 이런 전처리가 필요할 수 있겠다고 느꼈다.

### 3. 정규 표현식의 수량자와 경계

`04_regular_expression.ipynb`를 통해 `?`, `*`, `+`, `{n}`, `{min,max}`가 바로 앞 문자 또는 그룹의 반복 횟수를 정한다는 것을 확인했다.

- `?`: 0번 또는 1번
- `*`: 0번 이상
- `+`: 1번 이상
- `{n}`: 정확히 n번
- `{min,max}`: 최소와 최대 횟수 지정

특히 `?`는 항상 같은 역할을 하는 기호가 아니라 문맥에 따라 달라진다. `ab?c`에서는 `b`가 선택 사항이라는 뜻이지만, `(?:Mon|Tues|Fri)`에서는 비캡처 그룹을 만드는 문법의 일부다.

### 4. 그룹, 캡처, 역참조

괄호 `()`는 패턴을 묶는 동시에 매치한 값을 그룹으로 기억할 수 있다. 예를 들어 `(Mon|Tues|Fri)day`는 첫 번째 그룹에 `Mon`, `Tues`, `Fri` 중 하나를 저장한다. `findall()`에서는 캡처 그룹이 있으면 전체 문자열이 아니라 그룹 안의 내용만 반환할 수 있다. 그래서 단순히 묶기만 원할 때는 `(?:...)` 비캡처 그룹을 사용한다.

역참조 `\1`은 첫 번째 그룹에 기억한 값을 다시 사용하는 기능이다. HTML 예제의 `<(\w+)[^>]*>(.*)</\1>`에서는 여는 태그의 이름을 첫 번째 그룹에 저장하고, `</\1>`로 같은 이름의 닫는 태그만 찾는다. 예를 들어 `<div>...</div>`는 매치되지만 `<div>...</b>`는 매치되지 않는다.

치환 문자열의 `\2`는 두 번째 그룹을 뜻한다. 따라서 `re.sub(pattern, r"\2", text)`는 태그 전체를 태그 내부 내용으로 바꾼다. 반복 치환을 통해 단순한 HTML 예제에서 태그를 하나씩 제거할 수 있었다.

## 새롭게 알게 된 점

정규 표현식은 기호를 외우는 것보다, 각 기호가 "무엇을 기억하는지", "몇 번 반복되는지", "문자열의 어느 위치를 검사하는지"를 따라가며 읽는 것이 더 중요하다는 것을 알았다. 특히 `\1`, `\2`는 단순한 숫자가 아니라 앞의 괄호 그룹을 가리키는 번호라는 점을 이해하니 HTML 태그 처리 예제가 훨씬 자연스럽게 읽혔다.

또한 `\w+`는 "아무 글자"가 아니라 문자·숫자·밑줄 등의 단어 문자라는 점, `[^>]`는 `>`를 제외한 문자라는 점도 정확히 구분해야 한다.

그동안 게임을 만들면서는 이미 정리된 데이터나 API 응답을 받아 사용하는 일이 많아서, 그 데이터가 어떤 규칙으로 추출·검증되는지까지 해석할 필요를 크게 느끼지 못했다. 이번 정규 표현식 학습을 통해서는 화면에 보여 주기 전의 데이터도 코드로 안전하게 다룰 수 있어야 한다는 점을 알게 되었다.

자연어 처리에서는 표제어 추출, 특수문자 처리, 띄어쓰기와 형태 통일 같은 정규화가 필요한 경우가 많다. 실제 구현 코드는 GPT나 Codex에게 목적과 조건을 설명해 작성해 달라고 요청할 수 있다. 다만 도구가 만든 코드를 그대로 사용하기보다, 어떤 규칙으로 텍스트가 바뀌는지와 결과가 의도에 맞는지를 해석할 수 있어야 한다. 그래서 정규화와 정규 표현식의 기본 원리는 앞으로도 숙지할 필요가 있다고 느꼈다.
